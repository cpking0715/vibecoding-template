# 手动模型切换指南

本指南适用于 **Cursor / Claude Code / 其他不支持 Subagent 的 AI 工具**。
开发者根据当前 Phase 手动切换模型，以达到成本和质量的最佳平衡。

## 总原则

- **全程不切 ≈ T2（Sonnet）成本**：最省心，月 ~$20
- **只在 3 处切到 Haiku（T1）**：写测试、生成文档、修 lint，月 ~$8
- **只在 2 处切到 Opus（T3）**：架构设计、安全审查，月 ~$30
- **按表切 ≈ 本方案**：月 ~$12-15，比全程 Sonnet 省 30-40%

---

## 按 Phase 的切换表

### Phase 0: PRD + 设计方案

| 步骤 | 推荐模型 | 切换时机 | 说明 |
|------|---------|---------|------|
| Step 0a 写 PRD | **Sonnet** | 开始 Phase 0 时 | 默认模型，不需要切 |
| Step 0b 架构设计 | **→ Opus** | 进入 Step 0b 时切 | 架构决策影响整个功能，值得用最强模型 |
| Step 0b 确认后 | 切回 Sonnet | 用户确认设计方案后 | 后续 Phase 回到默认 |

**节省技巧**：如果你对项目架构已经很熟悉，全程 Sonnet 也可。

---

### Phase 0.5: 验收场景列表

| 步骤 | 推荐模型 | 切换时机 | 说明 |
|------|---------|---------|------|
| 生成场景列表 | **Sonnet** | 默认 | 不需要切 |

**不必切**：验收场景是结构化表格，Sonnet 完全够用。

---

### Phase 0.8: 并行评估（可选）

| 步骤 | 推荐模型 | 切换时机 | 说明 |
|------|---------|---------|------|
| 依赖图分析 | **Sonnet** | 默认 | 不需要切 |

**不必切**：分析工作量大但不复杂，Sonnet 足够。

---

### Phase 1: 需求拆解

| 步骤 | 推荐模型 | 切换时机 | 说明 |
|------|---------|---------|------|
| 拆解 Micro-step | **Sonnet** | 默认 | 不需要切 |

**不必切**：拆解是结构化的，Sonnet 足够。

---

### Phase 2: TDD 循环（关键省钱阶段）

| 步骤 | 推荐模型 | 切换时机 | 说明 |
|------|---------|---------|------|
| **2a** 验收标准 | **Sonnet** | 默认 | — |
| **2b** Red（写测试） | **→ Haiku** ✔️ | 进入 Step 2b 时切 | 测试代码是结构化输出，Haiku 质量已经足够 |
| 2b 确认测试失败后 | **切回 Sonnet** | 进入 Step 2c 前 | — |
| **2c** Green（CRUD） | **Sonnet** | 默认 | — |
| **2c** Green（核心逻辑） | **→ Opus** | 涉及核心算法时切 | 切到 Opus，改完切回 |
| **2c** Green（安全/支付） | **→ Opus** | 涉及安全逻辑时切 | 切到 Opus，改完切回 |
| **2d** Refactor | **Sonnet** | 默认 | — |
| **2e** lint/typecheck 修复 | **→ Haiku** ✔️ | 修复 lint 错误时切 | 简单的格式化修复 Haiku 足够 |
| 2e 修复完 | **切回 Sonnet** | 准备 commit 时 | — |

**节省关键**：一次 TDD 循环中，测试约占 30% token、lint 修复约占 10%。用 Haiku 做这两步，可节省 ~40% 的 TDD 成本。

---

### Phase 3: Agent Loop

| 步骤 | 推荐模型 | 切换时机 | 说明 |
|------|---------|---------|------|
| 第1-2次修复 | **Sonnet** | 默认 | — |
| 第3次修复 | **→ Opus** | Sonnet 修 2 次仍失败时 | 升级到最强模型 |
| 修复通过后 | 切回 Sonnet | — | — |

**节省关键**：80% 的失败在第 1-2 次就修好了，用 Opus 的概率很低。

---

### Phase 3.5: 综合验收测试

| 步骤 | 推荐模型 | 切换时机 | 说明 |
|------|---------|---------|------|
| 数据库测试 | **Sonnet** | — | 脚本执行，不需 AI |
| 性能测试 | **Sonnet** | — | 执行工具，不需 AI |
| 功能 E2E 测试 | **Sonnet** | — | 不消耗 AI token |
| 异常测试 | **Sonnet** | — | 设计场景用 Sonnet |
| 安全测试（npm audit） | **Sonnet** | — | 执行命令，不需 AI |

**不必切**：Phase 3.5 主要是执行测试命令而不是生成代码，AI token 消耗很低。

---

### Phase 4: 安全 PR 门禁

| 步骤 | 推荐模型 | 切换时机 | 说明 |
|------|---------|---------|------|
| AI 扫描变更代码 | **Sonnet** | 默认 | — |
| 如果扫描到敏感代码 | **→ Opus** | 需要人工审核时 | 用最强模型分析安全风险 |
| 审核完成 | 切回 Sonnet | — | — |

**节省关键**：90% 的 PR 无敏感代码，不需要切。

---

### Phase 5: 文档生成（第二大省钱点）

| 步骤 | 推荐模型 | 切换时机 | 说明 |
|------|---------|---------|------|
| 生成 API 文档 | **→ Haiku** ✔️ | 进入 Phase 5 时 | 基于代码的静态分析，Haiku 足够 |
| 更新 README | **→ Haiku** ✔️ | 同上 | 同上 |
| 全部生成完 | 切回 Sonnet | 进入 Phase 6 前 | — |

**节省关键**：文档生成是纯文本总结，不需要复杂推理。用 Haiku 可节省～50% 的文档成本。

---

### Phase 6: 推送与合入

| 步骤 | 推荐模型 | 切换时机 | 说明 |
|------|---------|---------|------|
| 生成 PR 描述 | **Sonnet** | 默认 | 不需要切 |
| git tag | — | — | 不需要 AI |

**不必切**：主要是 git 操作，AI token 消耗可以忽略。

```
═══════════════════════════════════════════════
  阶段         切到          切回
───────────────────────────────────────────────
  Phase 0b      → Opus        完事切回 Sonnet
  Phase 2b      → Haiku       进 2c 前切回
  Phase 2c 核心  → Opus        完事切回
  Phase 2e lint → Haiku       修完切回

```

## Cursor 具体操作

| 操作 | 快捷键 |
|------|--------|
| 切换模型 | `Cmd+Shift+P` → "Switch Model" |
| 查看当前模型 | 状态栏右下角显示 |
| 默认模型设置 | Cursor Settings → Models → Default |

切换模型后**不会丢失当前对话上下文**，可以无缝继续。

## Claude Code 具体操作

```bash
# 切换到 Haiku
export ANTHROPIC_MODEL=claude-3-5-haiku-latest
claude

# 切换到 Opus  
export ANTHROPIC_MODEL=claude-3-opus-latest
claude

# 默认为 Sonnet（不需要设置）
claude
```

## 如果完全不想手动切换

**就用 Sonnet 跑全程**。比 Opus 便宜 5 倍，比全手动切 Haiku 省心，质量对于 90% 的任务已经足够。多出来的 10% 成本（约 $5/月）就当买省心。
